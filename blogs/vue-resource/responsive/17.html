    # nextTick

    # 异步更新队列 - nextTick()

    • Vue更新 DOM 是异步执行的, 批量的
      • 在下次 DOM 更新循环结束之后执行延迟回调。在修改数据之后立即使用这个方法, 获取更新后的DOM
    • vm.$nextTick(function() { /* 操作DOM */ }) / Vue.nextTick(function() {})

    # 案例展示

    
const vm = new Vue({
  el: "#app",
  data: {
    msg: 'hello nextTick',
    name: 'Vue.js',
    title: 'Title'
  },
  mounted() {
    this.msg = "Hello World";
    this.name = "Hello snabbdom";
    this.title = "Vue.js";
    this.$nextTick(() => {
      // * 进入此处后, 第一个tick已经执行完成, 也就是说mountComponent执行完成, mounted也已经走完了, 组件挂载完毕, 所以此时已经可以获取到 $refs 下的对象了
      console.info(this.$refs.p1.textContent);
    })
  }
})

    # 调用位置

    • 手动调用vm.$nextTick(() => {})
    • 在Watcher.update中调用queueWatcher, 然后在queueWatcher的最后, 调用nextTick(flushSchedulerQueue)
    • 上述的nextTick均来自src/core/util/next-tick.js

    # 源码分析

    1. nextTick函数接收两个参数, 一个是cb, 一个还是ctx
    • cb代表当前传入的回调函数
    • ctx一般情况下是vue实例, 也自定义可以改变其指向

    1. 函数内部首先定义了一个_resolve变量, 后续可以看到他被赋值, 这里只定义没有初始化

    2. 将cb加上异常处理, 并push到callbacks数组中, 这个数组内部存储了一系列回调函数

    • 这个cb是外部传入的, 所有外部传入的函数对于vue来说都是危险的, 所以会加上try catch
    • 如果cb没有传递, 那么就会判断_resolve是否被初始化, 如果有, 那么会调用_resolve

    1. _resolve的初始化在最后面, 返回了一个new Promise, 在promise的回调函数中, 将resolve赋值给了_resolve

    2. 接着判断pending, pending表示当前队列是否正在被处理

    • 如果没有被处理, 则将pending设为true, 标记为正在被处理
    • 然后通过timerFunc去处理callbacks中的回调函数
    1. timerFunc
    • 首先定义了一个timerFunc, 但不初始化, 根据后续的环境, 采用不同的方式初始化, 避开兼容问题
    • 大部分情况下都会走第一个判断, 也就是当前浏览器原生支持promise的情况, 优先使用Promise来处理
      • 给timerFunc初始化为一个箭头函数
        • 箭头函数内部首先会使用Promise.resolve.then去触发flushCallbacks
        • 然后判断是否IOS环境, 如果是, 则需要在微任务执行完之后添加一个宏任务执行一个空函数
        • IOS使用的是UIWebView, 在 IOS版本 >= 9.3.3 时不会直接使用Promise, 可能会有潜在问题, 所以会降低成setTimeout
      • 将isUsingMicroTask设置为true, 标记为使用微任务
      • 微任务是在本次循环中的同步任务执行完毕后, 才会执行微任务
      • 为nextTick的作用是获取dom上最新的数据, 当微任务执行的时候, dom元素还没有渲染。 虽然数据已经被改变了, 当数据改变时, 会立即发送通知, 最终通过renderWatcher去刷新视图, 并且在这之前, 其实virtual dom树已经被更新了, 只是没有更新到页面上, 更新到页面需要等待当前事件循环结束。而nextTick如果使用微任务, 那么当次循环其实还没有结束, 此时会直接从DOM树上获取数据, 虚拟dom还没有被渲染到浏览器上, 这一点很重要
    • 如果浏览器并不原生支持promise且不是IE, 同时浏览器原生支持MutationObserver, 则使用MutationObserver来创建微任务
      • MutationObserver的作用是监听DOM对象的改变, 当DOM对象发生变化后, 会执行传入的回调函数, 而这个回调函数也是以微任务的形式执行
      • 由于IE10以上才支持MutationObserver这个对象, 且在IE11还会有些小问题, 所以这里过滤IE
      • 一般是在PhantomJS, IOS 7 和 Android 4.4上使用
      • 首先初始化一个 counter并将值设置为1
      • 接着定义一个observer, 然后观察一个文本节点, 文本节点的初始文本就是counter
      • 接着定义timerFunc
        • 改变counter
        • 重置上面文本节点的值为新的counter
        • 这与就会触发observer的回调函数, 也就是flushCallbacks
      • 将isUsingMicroTask设置为true, 依然标记为使用微任务
    • 接下来继续降级, 如果连MutationObserver都不支持, 则判断是否支持setImmediate(只有IE和NodeJs支持, 性能比setTimeout好, 会立即执行, 而setTimeout等待会更久(最快需要4ms)), 支持则直接使用其触发flushCallbacks
    • 以上API都不支持的情况下, 最后使用setTimeout(flushCallbacks, 0)兜底
    1. flushCallbacks很简单, 就是将pending设置为false, 因为处理完了, 然后遍历callbacks数组一个一个执行
    const callbacks = []
/**
 * @params cb 传入的回调函数
 * @params ctx 指向当前vue组件的实例
 */
export function nextTick (cb?: Function, ctx?: Object) {
  let _resolve
  callbacks.push(() => {
    // * 使用try catch是为了不让回调函数的报错影响后续的执行
    if (cb) {
      try {
        cb.call(ctx) // * 执行传入的回调函数
      } catch (e) {
        handleError(e, ctx, 'nextTick')
      }
    } else if (_resolve) {
      // * 给Promise用的判断
      _resolve(ctx)
    }
  })
  if (!pending) {
    // * 只要没有进入过这里面, 那么pending一定是false
    pending = true
    // * 执行微任务函数, 实际上就是执行上面的callbacks数组中所有的函数, 也就是执行传入的cb
    // * 当然, 微任务是一个异步回调, 也就是说，他会在下一个tick，去执行cb
    // * 所以每一次到这里，都不会立即执行，而是等待这一次收集完成，timerFunc中的回调函数触发，才会去执行其中的 callback 数组下所有的任务
    timerFunc()
  }
  // $flow-disable-line
  if (!cb && typeof Promise !== 'undefined') {
    // * 如果没有回调函数，但是存在Promise对象, 那么就会将Promise的resolve 赋值给 _resolve
    // * 不传递cb, 那么this.$nextTick就会变成一个Promise对象，那么就可以使用 this.$nextTick.then(() => {定义函数})
    return new Promise(resolve => {
      _resolve = resolve
    })
  }
}

    let timerFunc
// * isNative表示浏览器原生支持, 这里首先要保证Promise可以使用，并且是原生支持的Promise
if (typeof Promise !== 'undefined' && isNative(Promise)) {
  const p = Promise.resolve()
  timerFunc = () => {
    p.then(flushCallbacks)
    // In problematic UIWebViews, Promise.then doesn't completely break, but
    // it can get stuck in a weird state where callbacks are pushed into the
    // microtask queue but the queue isn't being flushed, until the browser
    // needs to do some other work, e.g. handle a timer. Therefore we can
    // "force" the microtask queue to be flushed by adding an empty timer.
    if (isIOS) setTimeout(noop) // * 如果是IOS环境, 那么就在最后使用setTimeout这种宏任务去直接执行一个空函数
  }
  isUsingMicroTask = true // * 将是否使用微任务置为true
  // ? 如果以上Promise浏览器原生并不支持, 那么就会使用MutationObserver, 使用这个首先要排除IE，然后要判断IOS7.x版本下的MutationObserver
} else if (!isIE && typeof MutationObserver !== 'undefined' && (
  isNative(MutationObserver) ||
  // PhantomJS and iOS 7.x
  MutationObserver.toString() === '[object MutationObserverConstructor]'
)) {
  // Use MutationObserver where native Promise is not available,
  // e.g. PhantomJS, iOS7, Android 4.4
  // (#6466 MutationObserver is unreliable in IE11)
  let counter = 1
  const observer = new MutationObserver(flushCallbacks) // * 定义一个新的MutationObserver实例
  const textNode = document.createTextNode(String(counter)) // * 观察的DOM
  // * 观察的DOM为textNode, 观察DOM节点的  characterData  变化,  变化的时候, 就会执行 flushCallbacks

  // ? CharacterData 抽象接口（abstract interface）代表 Node 对象包含的字符。这是一个抽象接口，意味着没有 CharacterData 类型的对象。 
  // ? 它是在其他接口中被实现的，如 Text、Comment 或 ProcessingInstruction 这些非抽象接口。
  // ? 所以他监听的就是这个文本节点
  observer.observe(textNode, {
    characterData: true
  })
  timerFunc = () => {
    // * 取余
    counter = (counter + 1) % 2
    textNode.data = String(counter)
  }
  // ! 因此微任务函数的触发，就是依靠 counter 的变化去触发DOM节点上data的改变，data改变的同时, 就会触发flushCallbacks
  isUsingMicroTask = true // * 只有使用MutationObserver来触发回调微任务, 才会将 isUsingMicroTask置为true
} else if (typeof setImmediate !== 'undefined' && isNative(setImmediate)) {
  // * 如果上面两个都不支持, 就会使用setImmediate来执行flushCallbacks
  // Fallback to setImmediate.
  // Technically it leverages the (macro) task queue,
  // but it is still a better choice than setTimeout.
  timerFunc = () => {
    setImmediate(flushCallbacks)
  }
} else {
  // Fallback to setTimeout.
  // * 如果以上全部都不支持，就直接降级使用宏任务setTimeout来执行 flushCallbacks
  timerFunc = () => {
    setTimeout(flushCallbacks, 0)
  }
}

    function flushCallbacks () {
  pending = false
  // * 深拷贝 callbacks 数组第一层
  const copies = callbacks.slice(0)
  callbacks.length = 0 // * 清空callbacks数组
  for (let i = 0; i < copies.length; i++) {
    // * 将 callbacks 数组(备份版)遍历并执行一遍
    copies[i]()
  }
}

    # 总结

    nextTick的核心其实就是首先将传入的回调函数放入一个callbacks数组中, 然后优先以微任务的方式去触发这个callbacks数组中的方法

    如果浏览器不支持微任务, 就会逐渐降级, 最终降级为宏任务

    在queueWatcher的最后, 就会使用到nextTick去处理flushSchedulerQueue来执行watcher最终更新视图, 所以触发flushSchedulerQueue去执行watcher的update是异步的
